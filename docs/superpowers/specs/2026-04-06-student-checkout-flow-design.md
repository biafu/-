# 学生端下单闭环设计

日期：2026-04-06

## 1. 目标

在现有学生端首页、门店详情页、订单列表页基础上，补齐从加购到支付的完整下单闭环，并确保首版实现可以平滑升级到后续增强版能力。

本次设计优先实现以下目标：

- 学生可从门店详情页加入购物车并进入购物车页
- 学生可查看购物车、调整数量、删除商品，并按店铺发起结算
- 学生可在确认订单页完成正式试算并提交订单
- 学生可在支付结果页继续支付或跳转订单列表查看状态
- 前端状态和组件边界为后续地址切换、优惠券、备注、空态引导、跨店提示留好扩展位

## 2. 范围

### 2.1 本次实现范围

- 新增学生端购物车页
- 新增学生端确认订单页
- 新增学生端支付结果页
- 门店详情页新增“去购物车”入口
- 学生端导航新增购物车入口与数量提示位
- 接入正式接口：
  - `GET /api/student/cart`
  - `POST /api/student/order/preview`
  - `POST /api/student/order/create`
  - `POST /api/student/order/pay/{orderId}`
- 自动读取默认地址用于确认订单页提交

### 2.2 暂不实现

- 页内切换地址
- 优惠券选择
- 订单备注输入 UI
- 支付渠道选择
- 独立订单详情页
- 跨店统一结算

## 3. 方案选择

本设计采用“独立流程页方案”。

### 3.1 候选方案

1. 独立流程页：购物车、确认订单、支付结果分别使用独立路由
2. 抽屉式结算：在门店详情页内通过抽屉承载购物车和部分结算流程
3. 单页步骤流：在一个大页面中堆叠购物车、确认、支付结果

### 3.2 选择理由

选择方案 1，原因如下：

- 路由边界清晰，刷新、回退和状态恢复更稳定
- 页面职责单一，后续加地址、优惠券、备注不需要重构主流程
- 方便在确认订单页承载正式试算逻辑
- 支付结果页可单独处理跳转和状态展示，不污染其他页面

## 4. 页面与路由设计

### 4.1 新增路由

在学生端路由下新增：

- `/student/cart`
- `/student/checkout`
- `/student/payment-result`

### 4.2 页面职责

#### 购物车页

职责：

- 拉取正式购物车数据
- 按店铺分组展示商品
- 支持数量增减
- 支持删除商品
- 支持按店铺发起结算
- 对空购物车、多店商品进行清晰提示

#### 确认订单页

职责：

- 接收并恢复结算草稿
- 自动带出默认地址
- 调用正式试算接口
- 展示商品、配送费、优惠、实付金额
- 提交订单

#### 支付结果页

职责：

- 展示订单号、金额、状态
- 触发支付动作
- 提供“查看订单”和“返回首页”入口

### 4.3 现有页面调整

#### 门店详情页

- 保留加入购物车动作
- 新增“去购物车”入口
- 加购成功后可以继续停留当前页，不强制跳转

#### 学生端布局页

- 导航中新增购物车入口
- 预留购物车数量角标位置

## 5. 前端状态设计

新增一个轻量级学生结算状态模块，建议文件路径为：

- `src/stores/student-checkout.ts`

### 5.1 状态字段

- `cartGroups`
- `activeStoreId`
- `checkoutDraft`
- `previewResult`
- `submitting`
- `paying`

### 5.2 checkoutDraft 结构

第一版即使用完整草稿结构，避免后续升级时重写主流程。

```ts
type CheckoutDraft = {
  storeId: number
  items: Array<{
    skuId: number
    quantity: number
  }>
  receiverName: string
  receiverPhone: string
  receiverAddress: string
  userCouponId?: number
  remark?: string
}
```

### 5.3 设计原则

- 购物车页只负责编辑购物项，不直接拼装支付结果
- 确认订单页只消费 `checkoutDraft` 并基于正式接口试算
- 支付结果页不持有复杂编辑状态，只消费创建订单结果
- 与优惠券、备注相关的字段先保留在数据模型中，即使首版 UI 不展示

## 6. 接口映射设计

### 6.1 购物车接口

已存在：

- `POST /api/student/cart/add`
- `GET /api/student/cart`

前端补充能力：

- 封装购物车列表查询
- 若后端尚未提供增减/删除接口，则首版需评估是否补后端接口

购物车列表返回结构已知字段：

- `id`
- `storeId`
- `skuId`
- `productName`
- `skuName`
- `price`
- `quantity`
- `totalAmount`

### 6.2 订单试算与创建接口

已存在：

- `POST /api/student/order/preview`
- `POST /api/student/order/create`

请求体统一使用 `OrderPreviewRequest` 结构：

- `storeId`
- `receiverName`
- `receiverPhone`
- `receiverAddress`
- `userCouponId`
- `remark`
- `items`

其中 `items` 为：

```ts
type OrderItemRequest = {
  skuId: number
  quantity: number
}
```

试算返回用于确认订单页金额区展示：

- `goodsAmount`
- `deliveryFee`
- `discountAmount`
- `payAmount`
- `items`

创建返回用于跳转支付结果页：

- `orderId`
- `orderNo`
- `orderStatus`
- `payAmount`

### 6.3 支付接口

已存在：

- `POST /api/student/order/pay/{orderId}`

支付结果页调用成功后：

- 跳转订单列表或刷新订单状态
- 触发消息数量刷新事件

## 7. 数据流设计

主链路固定如下：

1. 用户在门店详情页点击“加入购物车”
2. 用户进入购物车页，拉取 `GET /api/student/cart`
3. 用户按店铺发起结算，页面生成 `checkoutDraft`
4. 确认订单页自动读取默认地址并补全收货字段
5. 确认订单页调用 `POST /api/student/order/preview`
6. 用户点击提交后调用 `POST /api/student/order/create`
7. 创建成功后跳转支付结果页
8. 支付结果页调用 `POST /api/student/order/pay/{orderId}` 或跳转订单列表

## 8. 组件拆分建议

为避免页面文件过重，建议抽出以下组件：

- `StudentCartStoreSection`
  - 单个店铺分组展示
  - 负责数量增减、删除、去结算按钮
- `StudentOrderAmountCard`
  - 金额摘要展示
  - 统一展示商品金额、配送费、优惠、实付
- `StudentCheckoutAddressCard`
  - 展示默认地址
  - 地址缺失时展示前往地址页的引导
- `StudentPaymentSummaryCard`
  - 展示支付结果页的关键信息

组件拆分原则：

- 展示型组件不直接发请求
- 页面级组件统一负责加载、跳转和异常提示
- 金额区组件只接收格式化后的展示数据

## 9. 异常处理与边界条件

### 9.1 空购物车

- 显示空态
- 提供“去首页逛逛”入口

### 9.2 未登录或角色错误

- 延续现有路由守卫
- 非学生身份访问购物车或结算页时跳回合适页面

### 9.3 默认地址缺失

- 确认订单页禁止直接提交
- 提示用户前往地址页创建地址

### 9.4 多店商品

- 购物车页按店铺分组展示
- 每个店铺单独结算
- 不支持跨店统一提交

### 9.5 金额或库存变化

- 如果试算失败，展示接口返回错误并停留在确认页
- 如果创建订单失败且原因是库存或金额变化，提示用户重新试算

### 9.6 支付动作

- 创建成功不自动支付
- 用户在支付结果页主动点击支付
- 支付成功后提供跳转订单列表入口

## 10. 升级到增强版的扩展位

本设计要求首版即为后续版本 3 预留扩展位。

### 10.1 地址切换

- 在确认订单页地址卡片区域预留“更换地址”动作位
- 当前仅跳转地址管理页，后续可替换为弹层或选择页

### 10.2 优惠券

- `checkoutDraft` 保留 `userCouponId`
- 金额区保留 `discountAmount`
- 后续只需新增优惠券选择 UI 与回填逻辑

### 10.3 备注

- `checkoutDraft` 保留 `remark`
- 后续仅需新增输入框与 preview/create 回填

### 10.4 空态与提示

- 购物车页结构要能承载“最近浏览门店”“去凑单”“跨店提示”等增强内容

## 11. 验收标准

- 学生可从门店详情页进入购物车并看到正式购物车数据
- 学生可按店铺发起结算并进入确认订单页
- 默认地址存在时，确认订单页可完成正式试算
- 学生可成功提交订单并进入支付结果页
- 学生可在支付结果页完成支付并在订单列表看到状态变化
- 空购物车、缺少默认地址、试算失败、创建失败时均有明确提示
- 状态结构已保留 `userCouponId` 与 `remark` 等增强字段

## 12. 实施注意事项

- 统一沿用当前学生端视觉风格，不另起一套设计语言
- 优先复用现有工具函数、消息事件与金额格式化模式
- 页面中的文本、状态和金额展示应与现有订单页口径一致
- 若发现购物车增减或删除缺少后端接口，需要在实施计划阶段明确补充范围

## 13. 风险与待确认事项

- 当前已确认存在购物车列表、订单试算、订单创建、订单支付接口
- 当前尚未确认购物车数量修改、删除、清空是否有后端接口支持
- 若缺少上述接口，本轮实现需在计划阶段决定：
  - 补后端接口并同步接入
  - 或在首版中限制购物车仅支持查看与去结算

本设计默认优先追求完整闭环；若后端接口缺失且补充成本较高，则需要在实施前重新收敛首版范围。
