# 新增表单承载方式优化设计

## 背景

当前项目中部分“新增 / 编辑”功能使用 `el-dialog` 承载，但表单复杂度已经超过了短弹窗的适用范围，导致用户必须频繁滚动鼠标才能看全信息，保存按钮也不总是处于稳定可见的位置。

这类问题主要集中在两处：

- 商家端商品新增 / 编辑
- 学生端地址新增 / 编辑

本轮目标不是重写业务流程，而是在尽量不改接口和路由结构的前提下，优化复杂表单的承载方式和填写体验。

## 目标

- 让复杂新增表单在首屏呈现更清晰
- 让用户在填写过程中始终能快速找到保存操作
- 降低“弹窗里滚套滚”的阅读和输入负担
- 保持现有接口、字段结构和主要交互逻辑不变

## 非目标

- 不将本轮表单改造成独立路由页面
- 不调整后端接口契约
- 不重构短表单场景
- 不修改商品、地址的业务校验规则

## 方案概述

采用“中改版”策略：

- 简单表单继续使用 `el-dialog`
- 复杂表单升级为 `el-drawer`

这样可以在不大幅改动结构的情况下，提升新增页的可读性、可操作性和空间利用率。

## 场景划分

### 保留弹窗的场景

以下场景继续保留 `el-dialog`：

- 新增分类
- 订单评价
- 管理员审核备注
- 结算页选择地址

这些表单字段较少，交互路径短，弹窗仍然是成本最低的承载方式。

### 升级为抽屉的场景

以下场景改为 `el-drawer`：

- 商家端商品新增 / 编辑
- 学生端地址新增 / 编辑

这两类表单都已经具备中长表单特征，适合切换为抽屉承载。

## 商家端商品编辑器设计

文件：

- `src/components/workspace/MerchantProductEditor.vue`
- `src/views/workspace/ProductsView.vue`

### 交互结构

- 右侧抽屉打开
- 头部展示标题和简短说明
- 主体区域独立滚动
- 底部固定操作栏，包含取消和保存

### 内容分区

商品表单按以下顺序拆分：

1. 基础信息
   - 商品类型
   - 分类
   - 名称
   - 图片
   - 描述
2. 规格信息
   - SKU 名称
   - 价格
   - 库存
   - 上下架状态
3. 套餐组成
   - 仅套餐类型显示
   - 固定套餐组成编辑器保持现有逻辑

### 体验要求

- 保存按钮始终可见
- 套餐组成区域不与底部操作栏发生遮挡
- 桌面端抽屉宽度优先控制在约 `720px`
- 移动端抽屉宽度自适应到更高占比

## 学生端地址编辑器设计

文件：

- `src/views/student/StudentAddressView.vue`

### 交互结构

- 地址新增 / 编辑从对话框切换为抽屉
- 采用单列卡片式布局
- 头部展示当前模式：新增地址或编辑地址
- 底部固定取消 / 保存操作

### 内容结构

表单顺序调整为：

1. 联系信息
   - 联系人
   - 手机号
2. 地址信息
   - 校区 / 楼栋 / 门牌等字段
   - 详细地址
3. 配送偏好
   - 标签
   - 默认地址开关
4. 地址预览
   - 用轻量预览卡展示组合后的地址结果

### 体验要求

- 不再依赖左右双栏才能展示完整信息
- 主体表单可滚动，但保存区固定
- 空间优先保证输入体验，不强调复杂装饰

## 通用布局规范

本轮形成一个统一的复杂表单布局规则：

- 复杂表单优先使用抽屉
- 抽屉内只有内容区滚动
- 头部与底部操作区固定
- 表单按业务分区展示，不做超长平铺
- 首屏优先露出高频字段

后续新增复杂表单时，优先复用这套模式。

## 实施顺序

1. 商家端商品编辑器由弹窗切换为抽屉
2. 学生端地址编辑器由弹窗切换为抽屉
3. 抽出可复用的复杂表单布局样式，减少重复 CSS

## 风险与注意事项

- 抽屉改造不能影响现有表单提交和编辑回填逻辑
- 商品编辑器包含套餐组成区，改造时要确保局部滚动和高度表现稳定
- 地址页当前已有自定义 UI，本轮需要保留已有视觉方向，只优化承载方式
- Element Plus 的抽屉默认样式可能需要补充自定义类，避免底部按钮被内容挤压

## 验收标准

- 商家商品新增 / 编辑不再依赖整块弹窗滚动才能完成填写
- 学生地址新增 / 编辑首屏信息层次更清晰
- 两处复杂表单都具备固定可见的取消 / 保存操作区
- 桌面端和窄屏场景下都能完整显示主要字段
- 现有新增、编辑、保存能力保持可用
